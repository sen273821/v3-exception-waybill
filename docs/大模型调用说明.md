# 大模型调用说明

## 一、使用定位

本项目中的大模型能力属于**可选加分项**，不参与主流程的强依赖执行。系统的核心原则是：

> **AI 只提供建议，不自动执行。**

即使大模型服务不可用，异常上报、扫描品控、审批流转、执行联动等核心业务流程仍应能够完整运行。

---

## 二、使用场景

当前设计中，大模型主要用于两个环节：

### 1. 异常分类辅助
在扫描录入或异常上报时，根据操作人输入的异常描述文本，对异常进行辅助分类，推荐：
- 异常类型
- 异常子类型
- 严重度等级

例如：
- 输入：“客户反馈收到货后箱体破裂，商品外观损坏”
- 推荐输出：`logistics / damaged / high`

### 2. 审批建议辅助
在审批详情页，根据当前工单信息与历史审批记录，为审批人提供：
- 建议是否通过 / 驳回 / 升级审批
- 建议对应的处理动作
- 建议依据说明

例如：
- 当前工单金额较高、异常类型为破损、历史相似案例多为升级审批
- 则 AI 可建议：“建议进入二级审批，并参考历史高金额破损单的处理方式”

## 三、使用模型

代码中已接入 `openai` SDK，预留了兼容 OpenAI API 的大模型调用能力。当前实现中可使用的模型来源包括：
- OpenAI 兼容接口
- DeepSeek 等兼容服务

在现有文档和代码说明中，项目以 `deepseek-chat` 作为示例模型来源。

## 四、Prompt 设计思路

### 1. 异常分类 Prompt
设计目标：让模型根据异常文本推荐结构化分类，而不是自由发挥。

核心约束：
1. 给模型明确的异常类型范围；
2. 要求输出结构化 JSON；
3. 要求附带置信度与解释；
4. 明确标注：该结果仅为建议，不得替代人工决策。

示例输出结构：

```json
{
  "type": "logistics",
  "subType": "damaged",
  "severity": "high",
  "confidence": 0.86,
  "explanation": "描述中包含破损、外包装受损等关键词"
}
```

### 2. 审批建议 Prompt
设计目标：不是只让模型说“建议通过/拒绝”，而是让它给出可追溯的理由。

核心约束：
1. 输入当前工单的关键摘要；
2. 输入相似历史审批记录摘要；
3. 输出建议结论；
4. 输出建议依据；
5. 明确引用了哪些历史记录或哪些条件触发该判断。

示例输出结构：

```json
{
  "suggestion": "upgrade_to_l2",
  "confidence": 0.78,
  "explanation": "当前金额高于一级审批常见区间，且历史相似破损工单多数进入二级审批",
  "references": [
    "历史工单 #A102",
    "历史工单 #B334"
  ]
}
```

## 五、风险控制策略

### 1. AI 建议不能自动执行
所有 AI 输出都必须在前端明确标注为：

> **AI 建议，需人工确认**

系统不会因为模型输出就自动：
- 修改工单状态
- 自动审批
- 自动生成赔付
- 自动执行库存动作

### 2. 必须给出依据，不能是黑箱结论
无论是异常分类建议还是审批建议，都必须附带解释信息，例如：
- 命中了哪些描述关键词；
- 参考了哪些历史记录；
- 为什么建议升级审批或赔付。

### 3. AI 服务失败不阻塞主流程
如果模型服务超时、失败或不可用：
- 扫描录入仍可继续；
- 异常上报仍可继续；
- 审批流转仍可继续；
- 系统退回纯人工判断模式。

这符合题目要求：**AI 能力是辅助，不应成为业务主路径的单点故障。**

## 六、当前实现情况说明

在当前项目代码中，大模型能力已经预留并接入了对应 API 结构，主要体现在：
- `/api/ai/classify`

但在本次本地测试中，验证重点仍然是：
- 扫描品控主流程

- 审批状态机
- 执行联动
- V2 接口降级能力
- 日志与监控

也就是说，本次交付中应将 AI 能力描述为：

1. **已预留并可接入；**
2. **属于可选加分项；**
3. **不会影响主流程完成。**

## 七、后续可优化方向

如果继续打磨 AI 能力，建议优先优化以下几点：

1. **增强自然语言容错能力**
   - 例如当前品控规则更适合“破损等级：2”这种结构化输入；
   - AI 可补足“外观破损、箱体破裂”这类自然语言识别。

2. **引入历史相似案例检索**
   - 在审批建议前先检索历史工单，再把检索结果传给模型；
   - 比直接让模型“凭空判断”更可解释。

3. **统一建议输出协议**
   - 不同 AI 能力统一输出 `confidence / explanation / references` 字段；
   - 方便前端和审计日志统一展示。

4. **增加 AI 调用日志**
   - 记录 prompt 摘要、模型名、耗时、成功失败状态；
   - 便于排查 AI 服务质量问题。
