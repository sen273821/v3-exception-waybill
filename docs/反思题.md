# 反思题

## 1. 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？

如果题目直接给出精确阈值，我仍然不会把规则硬编码在业务代码里，而是保留“配置化”的设计，只是初始默认值会更明确。原因是审批阈值和超时时长属于高频调整的运营规则，在真实业务中很容易因为组织架构、赔付成本、仓储成本变化而调整。

不同点主要在于：
1. 《需求理解与假设说明》文档会缩短，因为不需要自行补全这些留白；
2. 审批规则表、品控规则表中的默认种子数据可以直接采用题目给定值；
3. 测试用例会更稳定，因为边界值更清楚，可以直接围绕明确阈值设计临界测试。

也就是说，题目给出精确数值会降低需求理解成本，但不会改变我对“规则配置化”的架构选择。

## 2. 你认为本题里“审批状态变更”与“库存/赔付联动”之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理？

最大的风险点是：审批状态已经更新成功，但下游赔付记录或库存流水没有成功落库，从而出现“页面上显示已通过，但账务或库存没有跟上”的中间态。这类问题在异常处理系统里很危险，因为后续操作员通常会把“审批已通过”理解成“业务已执行完成”。

如果不用数据库事务，我会采用以下方案：
1. 先落审批事件，再异步执行下游动作；
2. 给审批动作生成唯一业务事件 ID；
3. 库存、赔付写入都基于该事件 ID 做幂等；
4. 如果下游失败，则通过补偿任务重试；
5. 页面状态不要直接写成“已完成”，而是区分为“执行中 / 已完成 / 执行失败待补偿”。

也就是说，不用事务时，我会把问题转成事件驱动 + 幂等 + 重试补偿的可靠执行模型。

## 3. 面对一句话级别的模糊需求，你倾向于“先问清楚再动手”还是“先做出一个版本再迭代”？为什么？

我倾向于：先识别关键不确定项，再动手做一个可运行版本。

因为如果完全不动手，只停留在提问阶段，容易迟迟没有结果；但如果完全不澄清就直接开发，也容易在错误方向上做很多无效工作。

更实际的做法是：
1. 先识别哪些问题会影响表结构、状态机、权限边界这类高成本决策；
2. 对这些关键问题做假设，并明确记录；
3. 先实现一个最小可运行版本；
4. 在测试或验收时，根据反馈快速修正。

这也是本题最重要的考点之一：不是等所有需求都完美清楚，而是在不完整信息下做合理工程判断，并把判断写清楚。

## 4. 如果产品经理后续要求新增第六种异常类型（比如“客户投诉服务态度”，不涉及货品本身、不涉及库存联动），你现在的表结构和状态机需要改动哪些地方？改动成本大不大？

改动成本不大，但前提是异常类型与执行动作是解耦设计。

我当前的思路是：工单表只保存 `type/subType/status`，真正的下游执行动作由规则或映射逻辑决定。这样新增“客户投诉服务态度”时，主要改动有：
1. 在异常类型枚举/配置中新增一个 subtype；
2. 在“异常类型 -> 执行动作”的映射中增加一条规则；
3. 如果该类型不涉及库存、不涉及赔付，则执行动作可以是“仅记录/人工回访/客服处理完成”；
4. UI 上增加筛选项与展示文案；
5. 测试用例中增加该类型的审批流验证。

因为它不涉及货品、库存、赔付，所以不需要重构核心表结构，属于扩展规则，而不是推翻架构。

## 5. 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个？你会怎么处理？

最先撑不住的，我判断大概率是以下三个环节：

1. 工单列表查询：如果按状态、类型、运单号、时间筛选，但没有合适索引，会先慢下来；
2. 超时任务扫描：如果 `/api/monitor` 或后台任务每次都全表扫描待处理工单，20 万条时成本会明显上升；
3. 跨系统同步日志表：如果每次 V2 调用都写日志，但没有归档和索引策略，日志查询页面也会变慢。

处理思路：
- 对 `status/type/externalCode/createdAt` 增加复合索引；
- 将超时处理改成基于 `status + createdAt` 的分页扫描，而不是全表扫描；
- 日志按时间归档，监控页只查最近 N 条；
- 列表接口使用分页和必要字段查询，避免一次查太多关联信息；
- 如果继续增长，再考虑读写分离或事件异步化。

## 6. 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 int 改成 decimal，你会如何处理 V3 侧的兼容？

我会遵循两个原则：接口向后兼容 和 字段类型宽容消费。

### 对外接口不破坏现有调用方
1. 新接口尽量增加在独立命名空间，例如 `/api/v3/*`；
2. 不修改 V2 原有 `/api/orders`、`/api/parse` 等既有接口语义；
3. 如果必须扩字段，只做“新增字段”，不删旧字段、不改旧字段含义；
4. 使用版本化思路，例如未来可扩成 `/api/v3.1/*` 或 header 版本协商；
5. 上线时先灰度验证，再切正式调用。

### 金额字段从 int 改成 decimal 的兼容
V3 侧不要假设金额永远是整数，而应该统一做：
- 入参解析时转成 `Number` 或更安全的金额对象；
- 数据库存储时优先保留浮点/十进制兼容能力；
- 规则判断时避免直接依赖字符串格式；
- 测试里增加整数、两位小数、边界值验证。

也就是说，V3 应该消费“金额语义”，而不是绑死在某一种字段类型上。
