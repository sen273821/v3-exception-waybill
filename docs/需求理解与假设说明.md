## 二、我的假设与设定

### ① 分级审批金额阈值

**设定：**
- 物流异常：`0 ~ 500` 元走一级审批，`> 500` 元走二级审批
- 品控异常：`0 ~ 300` 元走一级审批，`> 300` 元走二级审批

**依据：**
- 物流异常通常面向客户赔付，单笔风险较高；
- 品控异常更多面向供应商追偿，单笔金额通常低于客户赔付；
- 阈值不应硬编码，应存储在 `ApprovalRule` 表中可配置管理。

### ② 审批超时时长

**设定：**
- 待审批（pending）：24 小时
- 一级审批中（l1_approval）：12 小时
- 二级审批中（l2_approval）：8 小时

**依据：**
- 一级审批多由业务负责人处理，给 1 个工作日较合理；
- 越高级别审批对业务恢复影响越大，应给予更短时限；
- 超时时长以 `ApprovalRule.timeoutHours` 配置化管理。

### ③ 重提次数上限

**设定：**
- 最大重提次数：3 次
- 第 4 次仍被拒绝则自动关闭工单

**依据：**
- 允许上报人补充材料，但不能无限循环消耗审批资源；
- 本地测试已验证第 4 次关闭逻辑成立。

### ④ 物流异常类型映射（各类型对应的下游动作）

| 异常类型 | 下游动作 | 赔付方向 |
|---|---|---|
| lost（丢件） | 赔付客户 | 赔付给客户 |
| damaged（破损） | 赔付客户 | 赔付给客户 |
| rejected（拒收） | 退货入库 | 赔付给客户 |
| timeout（超时未签收） | 赔付客户 | 赔付给客户 |
| address_error（地址错误） | 重新发货 | 一般无赔付 |

**依据：**
- 丢件、破损、超时通常直接损害客户体验，应优先客户赔付；
- 拒收更接近逆向物流处理；
- 地址错误多为履约或录入问题，通常通过重发纠正，不必直接赔付。

### ⑤ 角色权限划分

| 角色 | 权限 |
|---|---|
| reporter | 上报异常、查看权限范围内工单 |
| approver_l1 | 一级审批 |
| approver_l2 | 二级审批 |
| qc_manager | 品控主管，可快速放行、查看品控工单 |
| admin | 全量查看与兜底操作 |

**依据：**
- 上报、审批、品控主管职责必须拆分；
- 题目明确要求上报人不能审批自己提交的工单；
- 品控主管的“快速放行”应属于高风险特权能力。

### ⑥ V2 数据同步频率与一致性策略

**设定：**
- 发起异常上报、扫描校验时：优先实时调用 V2
- 成功调用后：同步更新 V3 本地 `WaybillSnapshot`
- V2 不可用时：允许基于最近一次快照降级展示/处理，并在接口或页面标识缓存来源
- 总体策略：V2 为主数据源，V3 为业务快照与异常处理系统

**依据：**
- 题目要求关键动作必须真实校验运单存在性；
- 同时又要求 V2 故障时不能让 V3 全部不可用；
- 快照设计在一致性和可用性之间做平衡。

### ⑦ 品控暂扣超时时长（独立于审批超时）

**设定：**
- 品控暂扣超时：4 小时
- 超时后自动升级进入二级审批

**依据：**
- 品控暂扣本质上会造成压仓和出库延误，成本高于普通待审批；
- 因此必须显著短于 12~24 小时级别的审批超时；
- 4 小时符合仓内班次和异常处理节奏。

### ⑧ 品控规则触发阈值

**设定：**
- 数量差异 `>= 5%` 触发异常
- 破损等级 `>= 2` 触发异常
- 标签错误、规格不符、批次异常支持文本匹配扩展

**依据：**
- 数量差异小于 5% 可视为容忍误差；
- 破损等级 2 以上已经影响正常出库；
- 阈值不写死在代码中，而通过 `QcRule` 表管理。

**补充说明：**
当前实现对描述文本格式有一定依赖，例如需要输入“破损等级：2”才能稳定命中规则；像“外观破损、箱体破裂”这类自然语言目前不会自动推断为异常，后续可通过 AI 或关键词增强优化。

### ⑨ 品控主管角色权限边界

**设定：**
- 品控主管可执行“误判快速放行”
- 快速放行必须填写复核原因并留痕
- 品控主管不默认拥有一级/二级审批权限，除非兼任其他角色
- admin 作为兜底角色拥有同类能力

**依据：**
- 快速放行属于仓内高风险操作，不能开放给普通审批人；
- 如果品控主管天然拥有全部审批权，会导致角色边界模糊；
- 通过 admin 保留应急兜底能力。

## 四、当前实现与本地测试中的验证结论

在本地测试中，以下假设已经被实际验证：

- 低金额物流工单可由一级审批直接进入执行阶段；
- 高金额物流工单会在一级审批后升级到二级审批；
- reject 第 4 次会自动关闭工单；
- V2 宕机时，V3 仍可基于本地快照继续完成扫描与上报；
- 品控主管可以通过扫描记录执行快速放行；
- 同一运单同一 SKU 存在未关闭品控工单时，重复扫描只追加扫描记录，不重复建单。

同时也发现以下值得记录的问题：

1. `GET /api/monitor` 在 V2 宕机后有概率出现超时，说明监控面板的失败场景性能仍需优化；
2. 品控规则引擎对自然语言描述的容错性较弱，更适合结构化输入；
3. 并发审批时虽然能够阻止重复处理，但提示文案与题目期望的“该工单已被处理，请刷新”不完全一致。
