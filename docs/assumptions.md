# 需求理解与假设说明文档

## 一、项目概述

V3 运单全流程管理系统承接 V2 的录单输出数据，覆盖扫描品控、异常上报、分级审批、执行联动四个阶段。V3 与 V2 独立部署、独立数据库，通过 HTTP API 交互。

## 二、需求留白点及假设设定

### ① 分级审批金额阈值

**假设**：
- 物流异常：≤500 元走一级审批，>500 元走二级审批
- 品控异常：≤300 元走一级审批，>300 元走二级审批

**理由**：物流异常涉及客户赔付，单笔金额通常较高；品控异常多为来货质量问题，单笔金额相对较低。阈值通过 `ApprovalRule` 表可配置，不硬编码在代码中。

### ② 审批超时时长

**假设**：
- 待审批（pending）：24 小时
- 一级审批中（l1_approval）：12 小时
- 二级审批中（l2_approval）：8 小时

**理由**：越高级别的审批对业务影响越大，应给予更短的处理时限。超时时长存储在 `ApprovalRule.timeoutHours` 中，后台可调整。

### ③ 重提次数上限

**假设**：最多 3 次。

**理由**：防止工单在“拒绝-重提”循环中无限打转，3 次后自动关闭，促使上报人补充材料或走线下流程。

### ④ 物流异常类型映射

| 异常类型 | 下游动作 | 赔付方向 |
|---|---|---|
| 丢件 | 赔付客户 + 重新发货 | 赔付给客户 |
| 破损 | 赔付客户 | 赔付给客户 |
| 客户拒收 | 退货入库 | 赔付给客户 |
| 超时未签收 | 赔付/催派 | 赔付给客户 |
| 收货地址错误 | 重新发货 | 无赔付 |

**理由**：地址错误通常是内部录入问题，不涉及第三方赔付；丢件/破损/超时属于物流履约问题，需赔付客户。

### ⑤ 角色权限划分

| 角色 | 权限 |
|---|---|
| reporter | 上报异常、查看自己上报的工单 |
| approver_l1 | 审批一级工单 |
| approver_l2 | 审批二级工单 |
| qc_manager | 品控主管：快速放行、查看所有品控工单 |
| admin | 全部权限 |

**理由**：品控主管与审批人权限不重叠，快速放行是高风险操作，需单独角色。上报人不能审批自己提交的工单，防止自批自核。

### ⑥ V2 数据同步频率与一致性策略

**假设**：
- 扫描/上报时实时调用 V2 接口校验
- 同时更新本地 `WaybillSnapshot` 快照表
- V2 不可用时使用本地快照，并在 UI 标注“缓存数据，可能非最新”

**理由**：关键动作必须保证真实性；快照用于列表展示和降级。V2 数据变更但 V3 未感知时，以上次同步时间为准，差异由人工介入处理。

### ⑦ 品控暂扣超时时长

**假设**：4 小时，超时强制升级至二级审批。

**理由**：货物压仓产生仓储成本和出库延迟，品控暂扣超时应远短于审批超时。4 小时是常见的仓库班次切换周期。

### ⑧ 品控规则触发阈值

**假设**：
- 数量差异 ≥5% 触发异常
- 破损等级 ≥2 触发异常
- 标签错误、规格不符、批次异常按文本匹配触发

**理由**：阈值通过 `QcRule` 表可配置，初始值参考行业常见 AQL 抽检水平和破损分级标准。

### ⑨ 品控主管角色权限边界

**假设**：
- 品控主管可执行“误判快速放行”，绕过完整审批流程
- 快速放行必须填写复核原因并留痕
- 品控主管不能审批自己创建的工单
- 品控主管与一级/二级审批人权限不重叠

**理由**：快速放行是高风险兜底能力，必须可追溯、可审计。

## 三、如果允许，会向产品经理提出的问题

1. 分级审批的金额阈值是否因业务线/客户等级不同而变化？
2. 品控主管快速放行后，是否需要通知原审批人？
3. 赔付给客户 / 向供应商追偿的实际支付流程是否在本系统内完成？
4. 是否需要对接 WMS 真实库存系统，还是仅记录库存变更流水？
5. V2 接口鉴权密钥的轮换周期和安全策略是什么？
6. 多租户场景下，上报人跨仓库/商户的权限如何定义？

## 四、大模型调用说明

- 使用模型：DeepSeek `deepseek-chat`
- 调用点：
  - 扫描描述 → 推荐品控异常子类型/严重度（`/api/ai/classify`）
  - 历史审批记录 → 建议审批意见（`/api/tickets/:id/approve` GET）
- AI 输出必须包含 confidence 和 explanation，前端明确标注“AI 建议，需人工确认”
- AI 超时或失败不阻塞主流程
